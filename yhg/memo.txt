[크롤링]
데이터 수집시 "프로그램이 멈췄는데 처음부터 다시..." 같은 상황을 꼭 한 번은 겪게 됩니다. 아래 내용을 미리 고려해서 프로그램을 작성하면 그런 고통을 크게 줄일 수 있어요!

1. 예외 처리는 선택이 아니라 필수
크롤링이나 API 호출은 언제든 실패할 수 있습니다.

네트워크 오류: 일시적인 연결 끊김, 타임아웃
서버 응답 오류: 500 에러, 429 (요청 과다) 등
데이터 구조 변경: 특정 페이지만 HTML 구조가 다른 경우
예상 외 값: 빈 응답, 누락된 필드 등
하나의 요청이 실패했을 때 프로그램 전체가 멈추지 않도록, 반드시 try-except 구문으로 감싸서 에러가 나더라도 다음 요청으로 넘어갈 수 있게 만들어야 합니다. 에러가 발생한 URL이나 항목은 별도로 기록해두면 나중에 재시도할 수 있어요.

2. 중간 저장은 반드시!
수집량이 많을수록, 모아서 한 번에 저장하는 건 위험합니다. 중간에 프로그램이 꺼지면 그동안 수집한 데이터가 전부 날아갑니다.
CSV, JSON 등 파일 형태로 일정 건수마다(예: 50건, 100건) 주기적으로 저장하는 로직을 넣어두세요. 저장 주기는 데이터 양과 수집 속도에 따라 조절하면 됩니다.

3. 이어서 수집할 수 있는 구조 만들기 (재시작 로직)
프로그램이 중간에 꺼졌다가 다시 실행할 때, 처음부터 다시 수집하면 시간 및 비용 낭비입니다.
프로그램 시작 시 기존 저장 파일이 있는지 확인하고, 이미 수집된 데이터를 불러온 뒤 아직 수집하지 않은 페이지나 상품만 이어서 수집하도록 설계하세요. URL이나 상품 ID 같은 고유 식별자를 기준으로 수집 완료 여부를 판단하면 됩니다.

4. 비동기 수집으로 속도 높이기
수집할 페이지가 수백~수천 개라면, 하나씩 순차적으로 요청하면 시간이 매우 오래 걸립니다. Python의 비동기(async) 문법을 활용하면 여러 페이지를 동시에 요청할 수 있어요.
단, 동시 요청 수를 반드시 제한(Semaphore 등)해야 합니다. 너무 많이 보내면 IP 차단이나 서버 과부하가 발생할 수 있어요. 비동기가 어렵다면 순차 수집 + 중간 저장만 잘 해도 충분합니다!

5. 기타 팁

요청 간격 두기: 요청 사이에 1~2초 정도 대기 시간을 두어 서버에 부담을 주지 않도록 합니다.
로그 남기기: 진행 상황과 에러를 출력해두면 문제 발생 시 디버깅이 훨씬 쉬워집니다.
